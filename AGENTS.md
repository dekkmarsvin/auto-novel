# Auto-Novel Fork — Agent Context

This repository is a **fork** of [auto-novel/auto-novel](https://github.com/auto-novel/auto-novel).

## Critical Rules

### 1. ThemeGlossary Must Be Preserved

**ThemeGlossary (共用主題術語表) is the core reason this fork exists.**

The upstream has removed this feature entirely. When merging upstream updates, you MUST:

- **Never delete** ThemeGlossary-related files
- **Never remove** `themeGlossaryId` fields from models or DTOs
- **Never remove** `themeGlossaryRepo` dependencies from translate APIs
- **Restore** any ThemeGlossary code that upstream deletes during merge

#### ThemeGlossary merge invariants

When resolving upstream merge conflicts, verify the following are still true:

- `Application.kt` still registers `ThemeGlossaryRepository`, `ThemeGlossaryApi`, and `routeThemeGlossary()`
- `MongoClient.kt` still defines `MongoCollectionNames.THEME_GLOSSARY`
- Web and Wenku backend models still contain `themeGlossaryId`
- Web and Wenku API update-glossary bodies still accept `themeGlossaryId`
- WebNovelTranslateV2Api and WenkuNovelTranslateV2Api still inject `themeGlossaryRepo`
- Translation task creation still merges theme glossary terms before novel-specific glossary terms:
  `themeGlossary.glossary + novel.glossary`
- Web and Wenku frontend DTOs still contain `themeGlossaryId`
- GlossaryButton still lists ThemeGlossary choices and sends `themeGlossaryId` to web/wenku updateGlossary APIs

### 2. Domain Configuration

This fork uses `books.kotoban.top` as its production domain.
The upstream uses `n.novelia.cc`.

When merging upstream, replace all hardcoded `n.novelia.cc` references with `books.kotoban.top` in:

- `web/vite.config.ts` — `apiUrl` default and `/files-extra` proxy target
- `web/src/components/markdown/MarkdownView.vue` — `currentHost` fallback
- `web/src/util/useUserData/api.ts` — auth domain comment

Exception: `monitor.novelia.cc` is an upstream external service URL — do NOT replace it.

After merging upstream, run:

```powershell
.\scripts\check-fork-invariants.ps1
```

This script must pass before treating the merge as complete.

### 3. Docker Image Owner

This fork uses `dekkmarsvin` (not `auto-novel`) as the Docker image owner.
CI/CD workflows use `${{ github.repository_owner }}` to stay portable.

### 4. Dev Compose File

`docker-compose.dev.yml` is maintained in this fork for local development builds.
The upstream has deleted this file — do not delete it during merges.

### 5. Generated Frontend Declarations

The frontend auto-generated declaration files are ignored and should not be restored during upstream merges:

- `web/src/auto-imports.d.ts`
- `web/src/components.d.ts`

They are regenerated by the web build when needed.

### 6. Container Build Test

Use `docker compose -f docker-compose.dev.yml build web api` to verify local container builds.
The web image build runs the workspace build from the repository root, so `packages/crawler` and `packages/translator` are built before `web`.

### 7. GitHub Actions Status After Push

When an agent pushes commits to GitHub, it MUST check GitHub Actions status before reporting the push as complete.

Use GitHub CLI when available:

```powershell
$branch = git branch --show-current
$sha = git rev-parse HEAD
gh run list --repo dekkmarsvin/auto-novel --branch $branch --limit 10
```

If the pushed `HEAD` has queued or running workflow runs, wait for them and fail the task if CI fails:

```powershell
gh run watch <run-id> --repo dekkmarsvin/auto-novel --exit-status
```

If a pull request exists for the branch, prefer the PR-level check view:

```powershell
gh pr checks --repo dekkmarsvin/auto-novel --watch
```

If GitHub Actions cannot be read because `gh` is unavailable, unauthenticated, or the network/API is unavailable, report that explicitly in the final response instead of claiming CI passed.

## Upstream Merge Checklist

Use this checklist whenever syncing from upstream:

1. Before merging, note whether fork-only files are present:
   - `docker-compose.dev.yml`
   - ThemeGlossary files listed above
   - domain overrides using `books.kotoban.top`
2. Merge upstream and resolve conflicts without deleting fork-only functionality.
3. If upstream deletes ThemeGlossary code, restore the fork implementation and adapt it to any upstream API/model changes.
4. Replace upstream `n.novelia.cc` references with `books.kotoban.top`, except `monitor.novelia.cc`.
5. Do not restore generated frontend declarations:
   - `web/src/auto-imports.d.ts`
   - `web/src/components.d.ts`
6. Run `.\scripts\check-fork-invariants.ps1`.
7. Run the normal verification for the touched area. For container changes, run:
   `docker compose -f docker-compose.dev.yml build web api`.

## Architecture Overview

- **web/** — Vue 3 + Vite 8 frontend (Caddy serves static + reverse proxy to API)
- **server/** — Ktor (Kotlin) backend API
- **packages/crawler/** — Web novel crawler library (rslib)
- **packages/translator/** — Translation pipeline library (rslib)
- **daemon/** — Background tasks (Node.js)
